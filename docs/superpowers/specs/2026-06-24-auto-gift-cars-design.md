# 自动送车流程 设计文档

- 日期：2026-06-24
- 状态：待评审
- 作者：MysticEEEE（与 Claude 协作）

## 1. 目标

新增一个「自动送车」流程：自动进入游戏的礼物箱功能，筛选出「重复项」车辆，
把**所有没有「全新标记」的重复车**作为礼物送掉，用于批量清理车库。

带「全新标记」的车（newcartag）必须保留，**绝不能误送**。

## 2. 背景与现有可复用资产

本功能完全沿用现有图像识别流程的写法与基础设施，不引入新机制：

| 复用对象 | 位置 | 用途 |
|---|---|---|
| `enter_menu()` | `main.py:1800` | 进入主菜单（导航起点） |
| 跑图导航套路 | `main.py:1893-1912`（`logic_race`） | `enter_menu()` → `pagedown` 翻页 → `wait_for_image_gray(锚点)` 的导航模式，送车照此实现 |
| 制造商筛选套路 | `main.py:2446-2469`（`select_new_consumable_car_from_list`） | 点筛选 → 选某一项（`CCbrand.png`）的代码模式，送车用来选「重复项」筛选 |
| 全新标记识别 | `newcartag.png` + `find_image_with_element_multi` / `wait_for_image_gray` | F3 测试（`image_matcher.py:1877` `start_test_find_image`）已证明该模板可稳定识别全新标记 |
| 通用工具 | `game_click`/`hw_press`/`wait_for_image_gray`/`wait_for_any_image_gray`/`capture_region`/`check_pause`/`update_running_ui` | 点击、按键、找图、截图、暂停、进度 |

**AI/YOLO 说明**：现有 YOLO 模型（`models/fh6_car_select_yolo.pt`）含 `new`/`b600`/`car`
三类，其中 `new` 即全新标记。理论上可复用，但它在「选车界面」训练，礼物箱界面差异可能导致掉点。
**本功能 V1 不使用 YOLO，仅用模板匹配**（零训练成本、已验证）。YOLO 留作后续可选增强。

## 3. 流程设计

### 3.1 导航进入（复用现有模式）

```
enter_menu()                      # 复用：进入主菜单
→ pagedown 翻页若干次              # 复用跑图导航模式，翻到礼物箱所在页
→ wait_for_image_gray(礼物箱锚点)  # 定位并点击进入礼物箱
→ 点开筛选 → 选「重复项」           # 复用制造商筛选代码模式
```

进入后，列表即为全部「重复车」。

### 3.2 核心循环：逐卡「先验后送」

```
循环：
  1. 截取【当前选中卡】区域 → 模板匹配 newcartag.png 判断有无全新标记
  2. 若【有】全新标记 → 不送，按「右」移动到下一张卡，回到步骤 1
  3. 若【无】全新标记 → 执行「赠送 Enter 序列」（见 3.3）
  4. 赠送完回到选车界面，继续下一张
  直到满足停止条件（见 3.4）
```

**安全原则（最高优先级）**：全新检测是每次赠送 Enter 前的硬性闸门。
当识别结果不确定（低于阈值/截图异常）时，**一律按「有全新标记」处理 → 跳过不送**。
漏送（少送一辆重复车）可接受；误送（送掉全新车）不可接受。

### 3.3 赠送 Enter 序列（单辆车）

对应手动操作：
```
Enter (确认赠送)
→ Enter (送给谁，默认对象)
→ Enter (携带的话语，默认)
→ Enter (留下的名称，默认)
→ 等待转圈加载 1-3 秒
→ wait_for_image_gray(送出提示)   # 等「送出成功」提示出现，确认本次完成
→ Enter / 点击左下角按钮 跳过
→ 回到选车界面
```

各步之间使用固定 `time.sleep` + 关键节点用 `wait_for_image_gray` 等待画面就位，
避免按键过快导致错位。**不盲目连点**——「送出提示」必须等到，才认为一次赠送成功。

### 3.4 停止条件（三重保险）

1. **主信号（最可靠）**：赠送尝试后若出现「**当前车辆无法送出**」提示
   （`cannotgift.png` 之类），说明列表只剩正在使用中的车，已送完 → 停止。
2. **辅助信号**：画面只剩 **1 张车卡** → 停止。
3. **安全阀**：可配置「最大赠送数量」上限（`gift_max_count`，默认保守值如 200），
   防识别异常时无限循环点击。

> 备选思路（暂不采用为主逻辑）：开局先翻页统计车卡总数、倒数到 2 停止。
> 因翻页计数易跨页重复计错，故以「无法送出」提示为主停止信号，计数仅作安全阀。

## 4. 接入方式

- **GUI 按钮（主要）**：在界面新增「自动送车」按钮，手动点一下即运行本流程。
- **任务链（可选）**：像现有 `chk_1/chk_2/chk_3` 一样，提供一个可选配置项，
  允许无人值守时把送车加入任务链跑一次。默认不勾选。
- **不分配热键**。

## 5. 模块与边界

新增一个聚焦的流程方法（拟名 `logic_gift_duplicate_cars`），放入现有模块区
（`image_matcher.py` 或 `main.py` 的流程模块段，跟随现有组织）。其职责单一：
执行一次完整的送车流程。它依赖第 2 节列出的现有工具，不新增底层机制。

辅助函数（按需拆分，保持单一职责）：
- `navigate_to_giftbox()` — 导航进礼物箱并选「重复项」筛选
- `selected_card_has_new_tag()` — 判断当前选中卡是否有全新标记（安全闸门）
- `gift_current_car()` — 执行一辆车的赠送 Enter 序列，返回是否成功
- `is_gift_list_exhausted()` — 判断停止条件（无法送出提示 / 仅剩 1 卡）

## 6. 配置项（写入 config.json）

| 键 | 默认 | 含义 |
|---|---|---|
| `gift_max_count` | 200 | 单次最大赠送数量（安全阀，0=不限） |
| `gift_in_chain`（或复用现有 chk 机制） | false | 是否纳入任务链 |

## 8. 风险与缓解

| 风险 | 缓解 |
|---|---|
| 误送全新车 | 全新检测为硬闸门；不确定时默认跳过；「送出提示」必须等到才算成功 |
| 礼物箱界面与选车界面布局不同导致模板失配 | 用截图重新裁礼物箱专用模板；阈值单独标定 |
| 赠送后光标回流位置不定，导致漏卡/重复 | 以「右/下」固定方向推进；以「无法送出」提示作为确定性终止 |
| 连点过快错位 | 关键节点用 `wait_for_image_gray` 等画面就位，不盲目连点 |

## 9. 范围之外（YAGNI）

- 不做 YOLO 训练 / AI 模式（V1）。
- 不做「重复」标记的逐卡视觉识别（游戏自带筛选已处理）。
- 不做自定义赠送对象/话语/名称（全部走默认 Enter）。
- 自动抽奖功能为**独立的下一个项目**，单独设计，不在本文档范围内。
