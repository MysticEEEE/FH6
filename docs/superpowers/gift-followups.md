# 送车/抽奖 功能后续待办（按优先级顺序）

> 当前主线：先把送车 dry-run 测通 → 真送循环 → 抽奖。以下为已确认、排期靠后的增强项。

## 待办 1：左侧车辆信息面板验证（送车 + 专精加点 通用）

**目的**：在选车/送车界面用左侧信息面板确认"当前选中卡就是目标车款"，防误送/误选；
并天然识别跳过在用车 S2 834（其面板数值与 B600 目标车不同）。

**依据**：截图 `24左侧指定车辆的信息.png`（面板：马力/扭矩/车重/前轴/排气 + B600 + CR金额），
`25专精加点-设计与喷漆-车辆选择.png` 与送车界面用**同一个**左侧面板，随选中卡实时更新。
该面板是车款级唯一指纹，比缩略图匹配可靠得多（所有 IMPREZA 缩略图几乎一样）。

**做法**：裁目标车的**数值块**参考图（避开顶部会动的 logo/圆点），选车时截左侧面板区域模板匹配，
`conf>阈值` 才确认是目标车、才执行送出/加点。作为"动作前正向门槛"：① 无全新标记 ② 左侧面板匹配目标车。

## 待办 2：YOLO 模型重训（覆盖礼物界面 + 区分车款/在用车）

**目的**：让 `AI辅助` 的 YOLO 也能在礼物界面识别（当前模型在选车界面训练，礼物界面失效）。
非必需（模板已够用），属增强。

**步骤**：
1. 采数据（可半自动）：礼物 + 选车界面截图，覆盖 带/不带全新、不同位置/页、在用车 S2 834、两界面。约 150-300 张。可改 dry-run 自动批量存图。
2. 打标：`new`/`b600`/`car`（必要时加 `s2_834`/在用车）。半自动预标（现有模型 + 模板 newcartag/classB600/skillcar 生成框）→ 人工校正。
3. 训练：`ultralytics yolo train`，从现有 `best.pt` 微调，~50-100 epoch。
4. 验证 + 替换 `models/fh6_car_select_yolo.pt`。

**硬约束**：当前 torch 是 **CPU 版**（cuda 不可用）→ CPU 训练很慢。需 ① 装 CUDA 版 torch 用本地 N 卡，或 ② 云端（Colab/GPU 服务器）。**用户待定训练环境**。

**过拟合**：固定 UI 卡片识别，过拟合非主要风险；重点是数据代表性（覆盖各位置/页/两界面/在用车），留小验证集确认即可。

**用户需提供/决定**：① 训练环境（本地CUDA/云）② 采图（工具由 Claude 提供，用户跑游戏）③ 标注校正 ④ 是否加"在用车 S2 834"等类别。
